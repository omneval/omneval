# DuckDB-native catalog becomes the production default

Status: accepted (narrowly supersedes ADR-0004 §Catalog and ADR-0005's catalog-driver framing; all other decisions in ADR-0004 and ADR-0005 remain in force). The "Catalog durability via in-process backup, not Postgres" decision below is itself superseded by ADR-0007 — Catalog backup is now the operator's responsibility, not an Omneval-owned S3 mechanism.

ADR-0004 chose Postgres as the DuckLake Catalog because it was "the serialization point that makes multiple concurrent writers safe." ADR-0005 then made the Quack Server the *sole* process holding a direct Catalog connection — every Writer, Query API, and backfill process reaches the Lake only as a Quack client. That made Postgres's original justification moot: concurrent-writer safety is now provided by the Quack Server being the only writer, not by the Catalog backend. What Postgres still cost us: DuckDB-specific types get encoded as Postgres `VARCHAR` on the way in, the Catalog is an extra stateful dependency beyond the Metadata Store, and it is vulnerable to connection-pool pressure under concurrent Quack-client load. Definite.app's writeup on running DuckDB itself (served over the Quack protocol) as a DuckLake catalog (https://www.definite.app/blog/duckdb-quack-ducklake-catalog) reports the same trade discussed here and similar throughput gains. We decided to make the local DuckDB-file Catalog (`CatalogDriverLocal`, already implemented and spike-verified durable across restarts per `internal/lake/lakeserver/server.go`) the default Catalog backend for every profile, not just demo.

## Decisions bundled here

- **Default flip, decoupled from the Metadata Store driver**: `lakeserver.ConfigFromApp` previously inferred `CatalogDriver` from `Database.Driver` (the Metadata Store's backend) when `quack.server.catalogDriver` was unset — Postgres Metadata Store implied Postgres Catalog. That inference is removed. The Catalog backend and the Metadata Store backend are independent choices; the Catalog now always defaults to `duckdb` regardless of what the Metadata Store uses, unless `quack.server.catalogDriver: postgres` is set explicitly.
- **Postgres driver kept as an explicit opt-out, not deleted**: `CatalogDriverPostgres` remains fully supported code, reachable only via an explicit config override, for anyone who wants to lean on existing managed-Postgres HA/backup tooling instead of the mechanism below. It is expected to be a candidate for deletion later, once the DuckDB-file Catalog has proven itself in production over time — not as part of this change.
- **Catalog durability via in-process backup, not Postgres**: the Quack Server's Table Maintenance scheduler gains a new step that runs `CHECKPOINT` (flushing the WAL so the file is consistent to copy) and uploads the Catalog file to `s3://<bucket>/lake/catalog-backups/<timestamp>.duckdb`, pruning older backups beyond a retention count. Interval and retention are Helm values (`quack.server.backup.interval`, `quack.server.backup.keepCount`; defaults 1 hour / keep 24) rather than hardcoded, so operators can trade off cost against recovery granularity. This is the *only* DR mechanism for the Catalog now — there is no replication or standby.
- **Existing live instance migrates via rebuild, not replay**: there is no tool to replay Postgres's `ducklake_*` catalog tables into a fresh DuckDB Catalog file. The one production instance currently on `CatalogDriverPostgres` migrates by taking a maintenance window, standing up a fresh local-file Catalog, and re-registering the existing Parquet files as a new DuckLake table set. This discards DuckLake snapshot/time-travel history (not currently surfaced anywhere in the product) and accepts a bounded window of write downtime and potential data loss for any batches in flight at cutover. This is a one-time procedure, not a repeatable migration tool, because there is exactly one instance to migrate and no externally-hosted customers yet.
- **Concurrency validation closes ADR-0005's open risk**: ADR-0005 flagged "validation that DuckLake 1.5's inlined-insert/inlined-delete semantics work correctly over a `quack://` session" as an unresolved prerequisite, exercised only informally for the Postgres path. This ADR adds a permanent Go integration test in `internal/lake` (modeled on `TestDeleteProjectNoResurrection`) that drives a Quack Server on `CatalogDriverLocal` from multiple concurrent Quack-client writers plus a score-write-back client and a read-only query client, asserting no lost or corrupted writes. It needs no Docker/testcontainers, unlike the existing Postgres-path tests. The test asserts correctness, not throughput; no separate load-test infrastructure is introduced.

## Considered Options

- **Keep Postgres as the default Catalog (status quo)**: rejected — the concurrency justification from ADR-0004 no longer applies now that the Quack Server is the sole writer (ADR-0005), so the status quo pays Postgres's costs (type encoding, an extra stateful dependency, connection-pool exhaustion risk) for a benefit it no longer provides.
- **Online catalog migration (replay Postgres's `ducklake_*` tables into a new DuckDB Catalog file)**: rejected as bespoke, fragile engineering against DuckLake's internal catalog schema, which is not a stable public interface and would need re-validation on every DuckLake version bump — not worth it for a single instance with no externally-hosted customers.
- **No backup for the Catalog file ("rebuild from Parquet" as the permanent DR story)**: rejected — cheap periodic backup avoids turning every routine PVC hiccup into a maintenance-window event, at negligible cost.
- **Delete the Postgres driver outright**: rejected — it is working, already-tested code; removing it forecloses an escape hatch for stricter-HA self-hosted deployments for no benefit this cycle.

## Consequences

- `CONTEXT.md`'s "Catalog" and "Disaster Recovery" entries, written when Postgres was the only documented prod Catalog, are updated by this ADR.
- The Helm chart's `quack-server-statefulset.yaml` already provisions a PVC when `catalogDriver` is `duckdb`; it gains new `quack.server.backup.interval`/`keepCount` values and S3 write access for the Quack Server pod.
- `internal/lake/lakeserver` gains a new Table Maintenance step (checkpoint + backup + prune) and `lakeserver.ConfigFromApp` drops its `Database.Driver`-based inference.
- `internal/lake` gains a new concurrency integration test for `CatalogDriverLocal`.
- The existing live instance has a scheduled one-time downtime/rebuild cutover, tracked separately from this ADR.
- ADR-0004's "Catalog: Postgres" line and ADR-0005's "Postgres remains the documented prod catalog" framing are no longer authoritative — refer to this ADR for current Catalog-backend guidance. Every other decision in both ADRs (S3-first ingestion, the Batch Ledger, commit cadence, partitioning, Quack Server as sole gateway, Table Maintenance ownership, retirement of `internal/leader`) remains in force unchanged.
