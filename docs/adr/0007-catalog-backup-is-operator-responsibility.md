# Catalog backup is the operator's responsibility, not an Omneval-owned mechanism

Status: accepted (narrowly supersedes ADR-0006's Catalog Backup module decision; ADR-0006's `catalogDriver: duckdb` default and every other decision in ADR-0004/0005/0006 remain in force)

ADR-0006 made the local DuckDB-file Catalog the default and, to recover the "managed backups for free" property Postgres used to provide, added a Quack Server-owned mechanism that checkpoints the Catalog file and copies it to S3 on an interval, pruning old backups. Re-examining this: the Catalog file already lives on the Quack Server's `volumeClaimTemplate` PVC (`/data/catalog.duckdb`), and operators running Kubernetes in production almost always already have PVC backup practices in place — VolumeSnapshot policies, storage-layer replication, or equivalent — as part of running *any* stateful workload, not something specific to Omneval. Maintaining a second, Omneval-specific backup path to the same underlying data duplicates that effort and adds ongoing S3 storage cost for every operator, including those who never restore from it because their existing PVC backup tooling already covers the Catalog file along with everything else on the volume. We decided to remove the S3 backup mechanism entirely and treat Catalog backup as out of scope for Omneval, the same way we don't ship backup tooling for any other PVC-backed component.

## Decisions bundled here

- **S3 Catalog Backup module removed entirely**: `internal/lake/lakeserver/backup.go` (`RunBackup`/`RunBackupLoop`/`pruneBackups`) and its wiring in `services/quack/internal/server/server.go` are deleted, not just disabled by default. A deprecated-but-present feature that silently costs S3 spend if an operator forgets to disable it is worse than a clean removal.
- **Helm `quack.server.backup.*` values removed**: the `QuackBackupConfig` struct, its three env vars (`OMNEVAL_QUACK_SERVER_BACKUP_*`), the StatefulSet env wiring, and the corresponding `helm unittest` cases in `quack_test.yaml` are deleted. This is a breaking Helm-values change; the one live instance (already migrated to `catalogDriver: duckdb` with backup enabled per ADR-0006) needs an upgrade note telling it to stop relying on the removed values and put a VolumeSnapshot policy in place instead.
- **Pod eviction / OOM-kill / crash recovery needs no new code**: when the PVC survives (the normal case for a StatefulSet's `volumeClaimTemplate` on any network-attached storage class), Kubernetes reattaches the same PVC to the rescheduled pod and `lakeserver.Serve` simply reopens the existing Catalog file at its configured path — DuckDB replays its own WAL for crash consistency on open. This path already existed before this ADR; what's new is a permanent Go integration test (`internal/lake`, Docker-free) that kills a running Quack Server mid-write and asserts the restarted server recovers with no data loss, so this property is verified rather than assumed.
- **Lost/corrupted PVC recovery is the operator's job, with a documented fallback**: an operator who loses the Catalog PVC restores their own VolumeSnapshot (or equivalent). An operator with no such backup falls back to the standalone "Rebuild Catalog from Parquet" runbook (extracted from the old Postgres-cutover runbook's Phase 5), which re-registers existing Parquet files as a new DuckLake table set at the cost of discarding snapshot/time-travel history.
- **The old `docs/runbooks/catalog-cutover.md` is deleted, not archived**: it documented the one-time Postgres-to-DuckDB migration. Zero instances remain on `CatalogDriverPostgres`, so the migration procedure has no future reader; its only generally-useful content (the Parquet re-registration steps) moves into the new standalone runbook.

## Considered Options

- **Keep the S3 backup module, disabled by default**: rejected — a present-but-deprecated feature still has to be maintained, documented, and excluded from future refactors for no operator who would actually use it; a clean removal is less to carry.
- **Keep S3 backup as an optional extra layer on top of operator-managed PVC snapshots**: rejected — redundant backup of the same file via two different mechanisms is needless cost and complexity; operators who want defense-in-depth can already snapshot the PVC at whatever cadence they choose.
- **Shorten the backup interval instead of removing it (to address RPO concerns)**: rejected once it was established that the relevant failure mode (pod eviction/OOM with PVC intact) needs no backup-based recovery at all — the only scenario where the S3 backup mechanism would have mattered (PVC loss) is exactly the scenario operators' own PVC backup tooling already covers.

## Consequences

- `CONTEXT.md`'s "Disaster Recovery" and "Table Maintenance" entries, which described the S3 backup mechanism as the Catalog's durability story, are updated by this ADR.
- ADR-0006's "Catalog durability via in-process backup, not Postgres" decision is no longer authoritative; its `catalogDriver: duckdb` default and the concurrency-validation test it introduced are unaffected.
- The Helm chart loses `quack.server.backup.*` values; the live instance needs an upgrade note about putting a VolumeSnapshot (or equivalent) policy in place.
- `docs/runbooks/catalog-cutover.md` is deleted; a new standalone "Rebuild Catalog from Parquet" runbook takes its place as the documented last-resort recovery path.
