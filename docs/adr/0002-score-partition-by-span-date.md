# Scores in cold storage are partitioned by the span's start_time date, not the score's created_at

Cold storage uses Hive-partitioned Parquet at `s3://bucket/archive/project_id={id}/date={date}/`. Scores could be partitioned by the date they were *created* (when the Eval Worker finished) or by the date of the *span they annotate*. We chose the span's `StartTime` date. This keeps a score physically co-located with its span in S3, so a query joining spans and scores for a given time window reads only the relevant date partitions — it never needs to range-scan across a separate scores prefix to find scores that arrived late. The downside is that bulk-eval runs over historical spans write score files into old date partitions, but this is infrequent and the scan savings on every read query outweigh the occasional write scatter.
